附录（一）：什么是 Agent-Ready Manuscript（ARM）？

Agent-Ready Manuscript，简称 ARM，可以理解为：把一篇传统科研论文整理成一种既方便人阅读，也

方便 AI Agent 检查、调用、试运行和复用的结构化科研资产。

它不是新的论文格式，也不是简单把 PDF 转成 Markdown。ARM 的核心，是把论文中的**结论、证据、

方法、执行步骤、评测方式和来源记录**拆分成清晰、可关联的对象。


1. 为什么需要 ARM

普通论文主要为人类阅读设计。对于 Agent 来说，论文中很多关键信息分散在：

* 摘要和正文；

* 图表及图注；

* 方法章节；

* 补充材料；

* 参考文献；

* 代码仓库和数据链接。

Agent 即使能够“读懂”论文，也不一定能可靠回答：

* 论文提出了哪些核心结论？

* 每条结论由哪段文字、哪个图或哪张表支持？

* 哪些内容是作者原文，哪些是模型推断？

* 实验需要什么输入、工具和步骤？

* 哪些步骤可以直接执行，哪些必须人工确认？

* 如何判断复现是否成功？

* 论文及其处理过程能否被追踪和回放？




                                                      PAGE 13
NEURONCLAW / 七日 HACKATHON 赛题方案



ARM 就是为解决这些问题设计的。


2. ARM 通常包含什么

一个完整的 ARM 可以包含以下内容：


                  内容             作用

                  metadata       论文标题、作者、DOI、版本、许可等基本信息

                  claims         论文中的关键科学结论

                  evidence       支持或反驳每条结论的原文、图表和数据位置

                  protocol       对论文研究方法和实验过程的结构化描述

                  runbook        Agent 可以理解和执行的操作步骤

                  eval＿plan      如何验证运行结果是否符合预期

                  provenance     信息从哪里来、经过什么工具和模型处理

                  limitations    论文局限、缺失信息和不可自动执行部分

                  artifacts      代码、配置、运行结果、表格等相关产物




3. “Agent-Ready”具体意味着什么

“Agent-Ready”不代表 Agent 可以无条件自动复现实验，而是至少满足以下条件。

    1   可理解：论文内容被整理为明确的字段，而不是只有一整篇长文本。

    2   可追溯：每条结论都能够回到原文段落、图、表或补充材料。

    3   可检查：系统可以自动检查：

           是否缺少证据；

           引用是否完整；

           来源和许可是否明确；

           是否混入了模型自行补充的内容；

           执行步骤是否缺少输入或条件。



                                                          PAGE 14
NEURONCLAW / 七日 HACKATHON 赛题方案



        4   可执行：论文中的部分方法可以转化为带有输入、输出、工具和失败条件的步骤。

        5   可回放：能够查看 Agent 是如何解析论文、提取结论、关联证据和生成步骤的。

        6   可导出和复用：ARM 可以作为一个独立资产包，用于后续研究、知识图谱构建、实验设计或其他

Agent 任务。


4. 一个简单例子

假设论文中写道：

> 某种干预在动物模型中降低了海马炎症标志物，并改善了空间记忆表现。

传统论文中，这句话可能分散在摘要、结果、图表和讨论中。

转成 ARM 后，可能整理为：

```yaml

claim:

 id: C-001

 text: 该干预在当前动物模型中降低海马炎症并改善空间记忆

 type: experimental_result

 status: supported

```

对应证据：

```yaml

evidence:

 - source: Figure 3

      location: Results, Section 3.2

      relation: supports

 - source: Supplementary Table S2



                                                      PAGE 15
NEURONCLAW / 七日 HACKATHON 赛题方案



      relation: supports

```

对应方法步骤：

```yaml

runbook_step:

 action: 计算炎症标志物组间差异

 input: biomarker_table.csv

 tool: statistical_analysis

 expected_output: effect_size_and_p_value

 can_dry_run: true

```

对应限制：

```yaml

limitations:

 - 结果来自动物模型，不能直接推广到临床人群

 - 原始数据未完全公开

```

这样，Agent 不仅知道论文“说了什么”，还知道：

* 依据在哪里；

* 适用范围是什么；

* 哪部分能够验证；

* 哪部分不能自动执行。

附录（二）Agent 工作流构建


一个实用的 Agent 可以概括为六部分：


模型 + 指令 + 工具 + 运行循环 + 状态 + 安全与追踪


OpenAI Agents SDK 将 Agent 定义为“带有指令和工具的模型”，并提供工具调用循环、会

话状态、Guardrails、人工审批和运行追踪。


1. 先明确任务与输出


不要先设计多个 Agent。先写清楚：


      输入是什么；

      最终输出是什么；

      Agent 可以调用哪些工具；

      哪些情况必须停止或交给人工；

      如何判断任务成功。


以脑病证据整理 Agent 为例：


输入：公开、去标识化病例

输出：

- 支持证据

- 反证

- 证据缺口

- 风险标志

- review_required


建议使用 Pydantic 或 JSON Schema 定义输出，避免最终结果只是一段难以检查的文本。

4. 给 Agent 增加工具


Agent 与普通聊天模型的主要区别，是它可以根据任务调用工具。


from agents import Agent, Runner, function_tool


@function_tooldef search_literature(keyword: str) -> str:

    """在本地文献索引中搜索与关键词相关的论文。"""

    # Hackathon 中可以先使用确定性 fixture

    return (

           f"检索结果：与 {keyword} 相关的论文共 3 篇。"

           "包括一篇综述和两篇实验研究。"
    )



@function_tooldef validate_reference(reference: str) -> str:

    """检查引用是否包含 DOI、PMID 或其他稳定标识符。"""

    if "doi" in reference.lower() or "pmid" in reference.lower():

           return "reference_valid"

    return "reference_requires_review"


agent = Agent(

    name="Evidence Agent",

    instructions="""

    围绕用户问题组织文献证据。

    在需要文献时调用 search_literature。

    在输出引用前调用 validate_reference。

    不得将模型推断表述为外部证据。

    """,

        tools=[search_literature, validate_reference],
)

        agent,

        "查找轻度认知障碍相关的炎症机制证据。",
)

print(result.final_output)


Agents SDK 可以把普通 Python 函数包装为工具，并自动生成参数 Schema；也支持网页搜

索、文件搜索、代码执行和 MCP 等工具类型。


工具设计原则


每个工具应当：


         只完成一类明确任务；

         输入和输出结构清晰；

         返回真实结果，不返回虚构结果；

         有超时、异常和空结果处理；

         高风险操作需要人工审批；

         保留调用参数和返回结果。


不建议创建一个名为 do_everything() 的大型工具。

7. 增加安全规则

至少实现三类约束：


输入规则


      拒绝真实可识别患者信息；

      检查文件类型和大小；

      检查恶意指令或 Prompt Injection。


工具规则


      删除、发布、写数据库等操作需要审批；

      限制工具参数；

      只开放当前任务需要的工具。


输出规则


      不伪造引用；

      不把模型推断当成证据；

      不输出自主诊断或处方；

      证据不足时返回 review_required。


Agents SDK 的 Guardrails 可以对输入、输出和工具调用执行校验；高风险失败可以提前中

止运行。

多 Agent 什么时候使用


先构建单 Agent。只有角色确实不同，才拆成多个 Agent。


例如 Paper-to-ARM：


Orchestrator

├── 文档解析 Agent

├── Claim 提取 Agent

├── Evidence 对齐 Agent

├── Runbook 生成 Agent


                                                         PAGE 24
NEURONCLAW / 七日 HACKATHON 赛题方案


└── Validation Agent


常见组织方式有两种：


Manager 模式


主 Agent 保持控制权，将子任务分配给专业 Agent。


Handoff 模式


当前 Agent 将任务直接交给另一个 Agent，由后者继续执行。


OpenAI 官方文档将这两种方式分别称为 agents as tools 和 handoffs。


对于七日 Hackathon，建议：


      先完成一个 Agent 加两个工具的可靠闭环，再增加第二个 Agent。
